# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Working agreement
- **Never `git commit` or `git push`.** Make changes in the working tree only and leave all committing/pushing to the user, even after a task is complete and verified — do not commit unless the user explicitly asks in that moment.

## What this is

ArcaneAtlas ("Arcane Atlas (LCDnD)") is a PySide6 (Qt for Python) desktop app — a digital battle-map / virtual tabletop for in-person tabletop RPGs. The GM lays out maps and assets on a control window, and a separate **Player Window** mirrors a chosen region of that map onto a second screen/TV the players see. Supports static images, animated video maps (`.mp4`/`.webm`), and fog of war with reveal/hide tools.

- Python package, `.spec`, `.iss`: **arcaneatlas** (lowercase)
- Human-facing display text: **"Arcane Atlas"** (two words)

## Commands

Run from source (Linux):
```bash
./runlin.sh            # activates .venv and runs `python3 -m arcaneatlas`
# or directly:
python3 -m arcaneatlas
```
On Windows use `runwin.bat` (set `LOG_ENABLED=1` inside it to capture logs to `.\logs\`).

Regenerate the Qt UI after editing `mainwindow.ui` (required — `ui_mainwindow.py` is generated):
```bash
./editui.sh            # open mainwindow.ui in Qt Designer (pyside6-designer)
./build_ui.sh          # pyside6-uic ... --from-imports  (Linux)
# Windows: build_ui.bat   (note: omits --from-imports)
```

Build the standalone Windows app + installer (Windows only):
```bash
build_exe.bat          # clean PyInstaller build -> dist\ArcaneAtlas\
build_installer.bat    # Inno Setup installer from arcaneatlas.iss
```
Before a Windows release build, place static LGPL `ffmpeg.exe` + `ffprobe.exe` in `arcaneatlas/resources/bin/` (used for video codec conversion — see that folder's README and the "Video hardware decoding" architecture note). They're git-ignored and bundled via the resources `Tree` in the spec.

Regenerate the app icon (after editing `arcaneatlas/resources/icon.png`):
```bash
./build_icon.sh        # icon.png -> icon.ico, multi-res 16/24/32/48/64/128/256 (needs ImageMagick)
```
It writes a **multi-resolution** `.ico` in place; a single 256×256 entry looks wrong in small Windows contexts (taskbar/Explorer), hence the full size set.

There is no test suite or linter configured. CI is release-only (see below).

## Release automation (GitHub Actions)

`.github/workflows/release.yml` builds the **Windows installer** + **Linux tarball** and publishes them as assets on a GitHub Release. **The three apps (ArcaneAtlas, ArcaneEye, ArcaneAudio) each carry an identical copy of this workflow — keep them in sync; only the `env:` block at the top changes per app.** This is the whole release system; learn it here.

### The asset-name contract (why exact filenames matter)
The website (`arcanetools.org/_redirects`) links to `https://github.com/EricEngineering/<repo>/releases/latest/download/<exact filename>`. The workflow **must** attach assets with these exact names or the download buttons 404:
- `ArcaneAtlas-Setup-windows.exe` — Windows installer
- `ArcaneAtlas-linux.tar.gz` — Linux onedir tarball
- `ArcaneAtlas-macos.dmg` — macOS (unsigned)

These names carry **no version** (the version lives *inside* the installer, from `__init__.py`) — so `releases/latest` always resolves the same stable URL. ArcaneEye/ArcaneAudio use their own name prefix in the `env:` block.

### How to trigger a release — two ways, same jobs
1. **Push a version tag** (the canonical release):
   ```bash
   git tag v0.8.0
   git push origin v0.8.0
   ```
   The **`v` prefix is required** — the trigger is `tags: ['v*']`. The tag name becomes the release name.
2. **Manual run** — Actions tab → **Release** → **Run workflow**, enter the version (e.g. `0.8.0`). Good for testing without creating/moving a tag; it creates/updates the release for tag `v0.8.0` at the current default-branch commit.

Keep the tag and the code version equal: tag `v0.8.0` ⇢ `__version__ = "0.8.0"` in `arcaneatlas/__init__.py` (the single source of truth — the About dialog, the installer, and the website version badge all derive from it / the release tag; the tag is just the release label).

### Re-running the same version is safe (idempotent) — by design
Build jobs upload **artifacts**; a final `release` job collects them and publishes with `softprops/action-gh-release`, which **updates the existing release and overwrites same-named assets**. So re-running for the same version (e.g. after adding the macOS job, or fixing a build) just **replaces the assets** — no duplicate releases, no errors.
- Re-run via **manual dispatch**: just run it again with the same version. ✅ nothing to clean up.
- Re-run via **tag**: a tag can't be pushed twice, so move it — `git tag -f v0.8.0 && git push -f origin v0.8.0` (re-triggers, updates the release). Or delete the release+tag in the GitHub UI and re-push.
- The release is published as a **full release — NOT draft, NOT prerelease — on purpose**: GitHub's `releases/latest` endpoint (what the website uses) ignores drafts and prereleases, so either flag would make the download links 404. Don't set them.

### Build details (per platform)
- **Windows** (`windows-latest`): installs Inno Setup via Chocolatey (to the exact path `build_installer.bat` hardcodes), downloads a static **full** ffmpeg build into `arcaneatlas/resources/bin/`, then runs **`build_installer.bat`** — the *same script used locally* (PyInstaller build + Inno compile; version derived from `__init__.py`). The output `installer/output/ArcaneAtlas-Setup-<version>.exe` is copied to the contract name. (A PowerShell check fails the job if no installer was produced, since the `.bat` itself can exit 0 on some failures.)
- **Linux** (`ubuntu-latest`): installs the Qt/xcb system libs PySide6 needs *at build time* (PyInstaller's `collect_submodules` imports `arcaneatlas`, which imports PySide6), downloads a static ffmpeg build into `resources/bin/`, runs `pyinstaller --noconfirm arcaneatlas.spec` with `QT_QPA_PLATFORM=offscreen`, and tars `dist/ArcaneAtlas/` into the contract tarball.
- **ffmpeg is fetched in CI**, never committed (it's git-ignored — see the resources/bin note). The static builds **must include `libvpx` + `libwebp`** or the transparency/transcode path silently produces opaque/trailing output (see Video transcoding). If those download URLs ever rot, update `FFMPEG_WIN`/`FFMPEG_LINUX` in the workflow `env:`.
- **macOS** — a **universal2 (arm64 + x86_64)** build on the `macos-14` **Apple Silicon** runner, so the app runs on **both Apple Silicon and Intel Macs**. Critically it's built **entirely on the arm64 runner** — the deprecated/starved Intel `macos-13` runner is **never used** (that's what made us drop Intel originally; universal2 sidesteps it by cross-producing the x86_64 slice). Three things make universal2 work: (1) a **universal2 Python** installed from **python.org's `macos11.pkg`** (`actions/setup-python` is arch-specific, so it's replaced; a `lipo -archs` check confirms the interpreter is fat), (2) **universal2 wheels** — PySide6's macOS wheels are universal2, so `pip` on the universal2 Python pulls both slices, and (3) a **fat ffmpeg/ffprobe** — CI fetches the arm64 *and* amd64 martin-riedl builds and `lipo -create`s them into universal binaries in `resources/bin/` (still `libvpx`+`libwebp` verified). The spec sets `target_arch='universal2'` (macOS only) so PyInstaller emits both slices — which requires *every* collected binary to be fat; an arm64-only dependency wheel would fail the build with a clear "does not contain slices for all architectures" error. Then the spec's **platform-guarded `BUNDLE`** wraps it into `dist/ArcaneAtlas.app` and `hdiutil create` packages the `.app` + `/Applications` symlink into the `.dmg`. **The app icon (`.icns`) is generated in CI** — a *Generate app icon (.icns)* step runs the macOS runner's native `sips` + `iconutil` over the committed `resources/icon.png` (1024×1024) to produce `resources/icon.icns` right before the PyInstaller build; the spec's `BUNDLE` auto-picks it up (`icon=str(icns_path) if icns_path.exists() else None`). It's produced in CI (not committed) — same "make it in CI" approach as ffmpeg — because generating a proper Retina `.icns` needs Apple's `iconutil`. Without it the `.app` (and thus the DMG copy *and* the installed app) falls back to the generic Python rocket icon. The `.icns` is a data file in `Contents/Resources/`, not a Mach-O, so it needs no separate `codesign` (the app-wide signature covers it). **The DMG *volume* icon** (the mounted disk's own icon, vs. the app icon inside) is still generic — separate future polish needing an `.icns` + `hdiutil`/`SetFile`.
  - **The bundled ffmpeg tools are signed *before* the build** (the *Sign bundled ffmpeg tools* step, secret-gated): every Mach-O inside a notarized `.app` must carry a hardened-runtime Developer ID signature, and PyInstaller does **not** sign Tree-bundled data binaries — so `ffmpeg`/`ffprobe` (and any nested `.dylib`) are `codesign`ed first, then the build bundles them already-signed. Miss this and notarization rejects the app for an unsigned nested binary. Windows/Linux ffmpeg bundling needs none of this.
  - **martin-riedl.de is a community build service** (one maintainer's infrastructure), not an official Apple/FFmpeg source. It's stable-URL'd and reliable, and the `--enable-libvpx`/`--enable-libwebp` config grep guards correctness (a bad/missing build fails the job loudly, never silently ships) — so it's fine to depend on for now. **TODO(macos-ffmpeg-source): remove the third-party dependency** by either compiling ffmpeg from source in CI (slower build) or vendoring pinned arm64 `ffmpeg`/`ffprobe` binaries in the repo/release assets.
  - **CI asserts Gatekeeper acceptance** (the *Verify Gatekeeper would accept the app* step, secret-gated): after notarize+staple, `codesign --verify --deep --strict` + `spctl --assess --type exec` fail the build if the signed app wouldn't pass Gatekeeper — catching signing/notarization regressions in CI instead of on a user's Mac.
- **macOS signing/notarization is wired but secret-gated.** The `build-macos` job's steps *Import signing certificate* and *Sign, notarize & staple* are guarded by `if: env.MACOS_CERT_P12 != ''`. With the seven `MACOS_*` secrets present (below), the job imports the Developer ID cert into a throwaway keychain, PyInstaller signs every binary with a **hardened runtime + `packaging/entitlements.mac.plist`** (driven by the `MACOS_SIGN_IDENTITY` env the spec reads), and the `.dmg` is signed → `notarytool submit --wait` → `stapler staple`. **Without the secrets it degrades to an unsigned `.dmg`** (right-click → Open to bypass Gatekeeper) — so forks and the pre-secret state still build.
  - **Required secrets** (Settings → Secrets and variables → Actions; **never in the repo**): `MACOS_CERT_P12` (base64 of the Developer ID `.p12`), `MACOS_CERT_PASSWORD`, `MACOS_SIGN_IDENTITY` (`Developer ID Application: <name> (<TEAMID>)`), `MACOS_TEAM_ID`, `MACOS_NOTARY_KEY_ID`, `MACOS_NOTARY_ISSUER_ID`, `MACOS_NOTARY_KEY_P8` (base64 of the App Store Connect `.p8`).
  - **One cert + one API key cover all three apps** (they're tied to the Team, not the app) — add the *same* secret values to ArcaneEye/ArcaneAudio. The entitlements file needs the `allow-jit` / `allow-unsigned-executable-memory` / `disable-library-validation` keys or a PySide6 app fails to launch under the hardened runtime.
  - The `.p12`/`.p8`/`.key` source files live **outside the repo** — regenerate the cert (openssl CSR → Developer portal, **G2 Sub-CA**) when it expires (~5 yrs) and update the two `MACOS_CERT_*` secrets.
- **Windows code-signing (Microsoft Trusted Signing / "Artifact Signing") is wired but secret-gated.** The `build-windows` job **decomposes** what `build_installer.bat` does locally — it runs PyInstaller, then (if secrets present) signs `dist/ArcaneAtlas/ArcaneAtlas.exe`, then runs Inno Setup directly, then signs the produced installer — because the app exe must be signed **before** Inno bundles it. Both signing steps use `azure/trusted-signing-action` and are guarded by `if: env.AZURE_CLIENT_ID != ''`, so **without the secrets an unsigned installer is produced** (works, but SmartScreen shows "Unknown Publisher"). Only the launched `ArcaneAtlas.exe` + the installer are signed (2 signatures/build) — Windows Authenticode doesn't require every bundled dll signed (unlike macOS notarization), which conserves the Trusted Signing quota.

Symptom of missing/empty `AZURE_*` secrets or a skipped login: `DefaultAzureCredential … EnvironmentCredential not fully configured → run az login`.

## Architecture

### Module layout
Most application logic lives in **`arcaneatlas/main.py`** (~3400 lines: `MainWindow`, `Canvas`, `PlayerWindow`, the file-browser models, startup/diagnostics, and `main()` — the entry point, invoked via `arcaneatlas/__main__.py`). Three supporting modules were split out to keep `main.py` manageable; the split is deliberately shallow (few files, low abstraction):

- **`arcaneatlas/items.py`** — the graphics items for the shared scene: `InteractivePixmapItem`, `InteractiveVideoItem`, `AnimatedItem`, `PlayerViewItem`, `WelcomeItem`, plus the item helpers `grow_canvas_for()`, `_owner_window()`, `_mark_item_dirty()`. These reach the owning `MainWindow` **only at runtime** via `view.window()` — they never import `main`, so the graph stays acyclic.
- **`arcaneatlas/transcode.py`** — pure ffmpeg/ffprobe helpers (`_ffmpeg_tool`, `probe_video_codec`, `video_has_alpha`, `transcode_to_h264`, `transcode_to_animated_webp`, `_run_ffmpeg`, `_TranscodeSignals`, the `HW_VIDEO_CODECS`/`ALPHA_PIX_FMTS` sets). No dependency on `MainWindow`/`Canvas`/items.
- **`arcaneatlas/colorbutton.py`** — `ColorPickerButton` (kept separate because `ui_mainwindow.py` imports it).
- **`arcaneatlas/tokenizer.py`** — `TokenizerDialog` (round-token editor; see Tokens & the tokenizer). Imports only PySide6, never `main`.
- **`arcaneatlas/webserver.py`** — `WebServer` + `WebShareDialog` + `make_qr_pixmap` (LAN web sharing: streams the Player Window to browsers + lets remote clients drag GM-permitted tokens; see LAN web sharing). Imports PySide6 (`QtNetwork`/`QtWebSockets`/`QtWidgets`/`QtGui`) and lazily `qrcode`, never `main` — `MainWindow` constructs it and passes itself in.

**Import direction is one-way:** `main → items, transcode, tokenizer, webserver` and `main → ui_mainwindow → colorbutton`. Nothing imports `main` back. **Keep it that way** — the items/transcode/tokenizer/webserver modules must stay free of `main` imports (the runtime `view.window()` lookup / a passed-in window ref is how they talk to the window without a cycle).

**The circular-import constraint is narrow.** `Canvas` is *not* promoted in the `.ui` — `graphicsView` is a plain `QGraphicsView` swapped for a `Canvas` in code (`MainWindow.__init__`). The only cycle risk is if `ui_mainwindow.py` imported a class from `main.py` (a Qt Designer promotion) while `main.py` imports `ui_mainwindow`. As long as widgets that the `.ui` promotes live in their own module (like `colorbutton`), there's no cycle. If you ever move `Canvas` into its own `canvas.py`, you *could* then promote it in Designer and delete the manual placeholder-swap hack.

### Two-window, shared-scene model
The core design is **one `QGraphicsScene` shared by two `Canvas` (`QGraphicsView`) windows**:
- `MainWindow` — the GM's control view. Editing, tools, file browser, fog painting.
- `PlayerWindow` — receives the *same scene* (`Canvas.setScene(shared_scene)`); its panning/zooming/scrollbars are disabled so players see a fixed framing.
- `PlayerViewItem` (a `QGraphicsRectItem`, z=100) is a draggable rectangle drawn **only in the GM view** that defines which region of the scene the Player Window displays. `sync_player_view_to_camera()` keeps the player window's framing matched to this rectangle via `fitInView(rect, KeepAspectRatio)` — **keep `KeepAspectRatio`**; stretching to fill would distort the grid/tokens. The rect is sized `playerDisplay{Width,Height}×72` scene-px (72px = 1 inch), so true physical 1-inch squares require `playerDisplayHeight` (the usually-constraining dimension) to match the real panel. If the rect's aspect ≠ the player window's viewport aspect, `fitInView` letterboxes and the player sees *extra* surrounding map. To prevent that, in **auto mode** (`player_dims_auto`, default on) the non-anchor dimension is derived from the anchor × the selected screen's pixel aspect (`_refresh_auto_dims()` / `_selected_screen_aspect()`). The anchor (`player_dims_anchor`, "width" or "height") is whichever dimension the user last typed in the dimensions dialog — either can drive; the other follows. Pixel aspect (`QScreen.geometry()`) is reliable; `QScreen.physicalSize()` (EDID) is **not** for TVs over HDMI, so physical scale stays a manual measurement of the anchor dimension. The dialog shows the detected aspect and disables Auto when none can be determined (`_selected_screen_aspect_label()`).

Because GM and player share the scene, moving/adding items or revealing fog is reflected on both automatically — the work is in controlling *what the player sees* (framing + fog), not in syncing item state.

### Canvas auto-sizing & grid
The canvas is **not a fixed size** — it auto-sizes to fit the placed maps so load/memory cost stays proportional to the actual content (a single map ≈ base size; a large tiled dungeon grows as needed).

- `Canvas.canvas_rect` is the **single source of truth** for canvas size; the grid lines and the `sceneRect` derive from it. (Fog no longer depends on canvas size — its reveal path is in scene coords; see Fog of war.) `grid_size = 72px` = one 1-inch / 5-ft square.
- `Canvas.update_extent()` recomputes `canvas_rect` from `_base_rect()` (the `BASE_EXTENT`=5000 floor, centered on origin) ∪ `_content_rect()` (bounding rect of the `InteractivePixmapItem`/`InteractiveVideoItem`/`AnimatedItem` maps) + `CONTENT_MARGIN`, snapped outward to grid spacing. It **grows eagerly** but **shrinks lazily** — a side is only reclaimed once it exceeds the target by more than `SHRINK_SLACK` (hysteresis to prevent thrash while editing). It never shrinks below base and only ever trims empty space.
- `MainWindow.refresh_canvas_extent()` is the entry point: it calls `update_extent()`, and if the canvas changed, reapplies grid visibility/z-order and realigns fog. **Call this after any map is dropped, moved, resized, removed, or loaded** — the existing hooks are in `Canvas.dropEvent`, the item `mouseReleaseEvent`s (via `grow_canvas_for`), the "Remove" context actions, `InteractiveVideoItem._on_video_size`, and `open_map`.
- `reset_extent()` returns the canvas to base size (used at the start of `new_map` / `open_map`, before fitting to the loaded maps).

### Layers & z-ordering (backgrounds < objects < tokens)
Map items are partitioned into the three asset categories as **z-bands**, strictly enforced: a background can never cover an object or token, a token never sits under either. Within a band, items keep their own relative order and can be freely reordered.

- **Z-bands** (`LAYER_Z_BASE` in `main.py`): `backgrounds` base `0`, `objects` `1_000_000`, `tokens` `2_000_000` (bands huge enough for any realistic item count). Reserved z's sit outside every band: `GRID_BELOW_Z = -1`, `GRID_ABOVE_Z = 3_000_000`, `PLAYERVIEW_Z = 9_000_000`. **`PlayerViewItem` hardcodes `9_000_000` in `items.py`** (it can't import `main`) — keep the two in sync.
- **`MainWindow.restack_layers()` is the single enforcer.** It groups `_map_items()` by `_item_layer(it)`, sorts each group by current z (stable → preserves order), and reassigns `base + i`; then reasserts grid (`_restack_grid()`) and player-view z. **Call it after any add / load / reorder.** `_item_layer(it)` resolves the layer from `it.asset_category`, falling back to the category prefix of `asset_filename`, then `DEFAULT_DROP_CATEGORY`. `_map_items()` = scene items that are `InteractivePixmapItem`/`InteractiveVideoItem`/`AnimatedItem` (excludes grid/fog/welcome/player-view).
- **"Bring to Top" is layer-capped.** The per-item `bring_to_top()` in `items.py` just delegates to `MainWindow.bring_item_to_layer_top(item)`, which raises the item only above its *same-layer* siblings, then restacks. (The old behaviour raised above everything — don't reintroduce a global max-z bump.)
- **Grid placement is band-based.** `on_grid_above_toggled` → `_restack_grid()` snaps the grid group to `GRID_ABOVE_Z`/`GRID_BELOW_Z` (no more scanning item z-ranges). `refresh_canvas_extent()` reasserts it after a grid rebuild.
- **The Layers tab** is a `LayersTreeView` (`QTreeView` subclass) over a `QStandardItemModel`: three group rows (tokens shown on top), each child = a map item by **friendly asset name**, top-of-stack first. `update_layers_list()` rebuilds it (call after any scene change — already hooked in place/remove/Delete/load/reorder). Selecting a row selects the matching scene item (`_on_layer_row_selected`). Dragging a child reorders z **within its layer**; `LayersTreeView.dropEvent` defers to `MainWindow._commit_layer_reorder()` (via `QTimer.singleShot(0, …)`, after the model finishes mutating), which recomputes z from the new row order and **reverts cross-layer drops**.
- **Why child rows store a string key, not the item:** an internal drag-move makes `QStandardItemModel` *serialize* the row (QDataStream), which silently drops a stored Python/`QGraphicsItem` object. So each child stores `str(id(item))` (Qt.UserRole) — serialization-safe — mapped back to the live item via `self._layer_item_by_key` (rebuilt every `update_layers_list`). `_building_layers` guards selection signals during the rebuild.
- **Persistence:** each item's `z` is saved in the map JSON (alongside `type/asset/pos/size/rot`) and restored on load; `open_map` sets `asset_category`/`z` per item then calls `restack_layers()`. **Legacy maps without `z`** load every item at `z = 0`, so `restack_layers()` falls back to JSON array order within each layer — no migration needed.

### Per-item player visibility (hide from players)
Each map item has a **`visible_to_player`** flag (class attribute default `True` on `InteractivePixmapItem`/`InteractiveVideoItem`/`AnimatedItem`). When `False`, the item is **drawn in the GM view but skipped in the player view** — there's no per-view opacity in Qt, so this is decided *per paint pass*, the same trick `PlayerViewItem` uses.

- **How a paint pass knows which window it's in:** `Canvas.is_player_view` is a class attribute `False`, set to `True` only on the PlayerWindow's canvas (`PlayerWindow.__init__`). In each item's `paint()`, `items._painting_in_player_view(widget)` checks `widget.parent()` (the view) for that flag. If hidden **and** painting in the player view → `return` before drawing content; if hidden and painting in the GM view → draw content, then `items._paint_hidden_marker()` overlays a **red wash + thick diagonal stripes + bold border** so the GM can't miss it (content still shows through). The stripes are drawn by hand in **scene units** (not a `QBrush` hatch pattern, whose 1px cosmetic lines vanish when zoomed out) so they stay visible at any zoom — tune `STRIPE_W`/`STRIPE_GAP` there. The Layers tree also appends `· hidden` to the row name.
- **Toggle:** each item context menu has a "Hide from Players"/"Show to Players" entry (label flips with state) → `items._toggle_player_visibility(item, view)`, which flips the flag, `item.update()`s (repaints both views), updates the player viewport, refreshes the Layers tree, and marks dirty. **Batched over multi-selection:** if the right-clicked item is part of a multi-selection (`item.isSelected()` and >1 selected media items), the SAME new state (derived from the clicked item) is applied to *every* selected `InteractivePixmapItem`/`InteractiveVideoItem`/`AnimatedItem` at once. The GM canvas enables `RubberBandDrag` (`Canvas.__init__`; `PlayerWindow` forces `NoDrag`) so a left-drag on empty canvas marquee-selects; Ctrl+click also multi-selects.
- **Persistence:** saved as `visibleToPlayer` per item in the map JSON; `open_map` restores it (default `True` for legacy maps). All three paint methods and all three context menus carry the same small block — keep them in sync if you add a fourth media item type.
- **Selection chrome is GM-only too.** Each item's `paint()` computes `in_player = _painting_in_player_view(widget)` once and gates its selection box + resize handles on `not in_player`, so players never see editing UI. For items that call `super().paint()` (pixmap/video), it also strips `QStyle.State_Selected` from `option` in the player pass to suppress Qt's *built-in* selection outline.

### Locking & copy/paste
- **Two independent locks.** `Lock Assets` (`lockmap_checkBox` → `_on_lock_map_toggled`) toggles `ItemIsSelectable`/`ItemIsMovable` on every **non-token** map item; `Lock Tokens` (`locktokens_checkBox` → `_on_lock_tokens_toggled`) does the same for **token** items only. A "token item" is `_is_token_item(it)` = `InteractivePixmapItem` with `is_token` True. Both are re-applied after `open_map` (loaded items) and a freshly placed item releases the lock for *its* category (`_place_asset`) so it's immediately movable. Tokens still grid-snap regardless (see Tokens → Always grid-snapped).
- **Lock-on-open.** `open_map` sets `lockmap_checkBox` to `bool(self._map_items())` when **`self.lock_on_open`** is on (default) — i.e. a **content-bearing** map opens with Lock Assets enabled (don't nudge an existing layout), while a fresh/empty map (and `new_map`) stays unlocked. The behavior is a persisted setting (`lockOnOpen` in `settings.json`; a checkbox in the **Settings dialog** → `_on_lock_on_open_toggled`, backed by `self.lock_on_open`). Dragging/importing a new asset still unlocks (`_place_asset`).
- **Copy/paste canvas items** (`Ctrl+C`/`Ctrl+V`, `QShortcut`s in `__init__`). `copy_selection()` serialises the selected media items to JSON under the private MIME type **`CLIP_MIME`** (`application/x-arcaneatlas-items`) on the OS clipboard. `paste_clipboard()` pastes at the cursor (`_paste_scene_pos()` — mouse-over-canvas point, else view centre) with this precedence: **(1)** our MIME → `_paste_internal_items` clones each entry (`_item_from_clip_entry`, mirrors `open_map`'s per-type build), translating the group so its bounding-box centre lands at the cursor, new items selected; **(2)** else a clipboard **file** (`hasUrls`) or **image** (`hasImage` → temp PNG via `_write_clipboard_image_tmp`) → `_place_asset(...)`, i.e. a fresh **asset import** (category dialog / tokenizer), same as a drag. Putting in-app copies on the OS clipboard makes **last-copy-wins** fall out for free (an external image copied after an in-app copy takes over, and vice-versa). Paste auto-creates a map if none is open (like drop). Clipboard entries carry `asset`/`srcPath` so a not-yet-imported item (only `asset_path`) still pastes and imports on save.

### LAN web sharing (browser player view + token drag)
`arcaneatlas/webserver.py` (`WebServer`) lets browsers on the same Wi-Fi **see the Player Window** and **drag GM-permitted tokens** — the cheap pixel-stream + state-sync first cut of the "remote clients" future direction. **LAN-only, no port forwarding** (in-person table). The GM opens it via the **"Web" toolbutton** (`web_toolbtn.clicked` → `_open_web_share_dialog`) → a `WebShareDialog` with a Start/Stop button, the join URL `http://<lan-ip>:<port>/`, and a **QR code** for players to scan. Start/stop go through `MainWindow._start_web_sharing()` / `_stop_web_sharing()` / `_web_sharing_active()`. Firewall: the GM must allow **both `<port>` and `<port>+1` TCP** (page + WebSocket) — the dialog carries a **firewall notice** (showing the live ports) plus a **`FirewallHelpDialog`** ("Firewall Help…") with per-OS, novice-friendly guidance: a button that opens the native firewall UI (`_open_os_firewall_settings`: `control firewall.cpl` / the macOS Firewall settings URL / a Linux firewall GUI), and on Linux a **copy-paste command** auto-targeted at the active firewall (`_linux_firewall_command` probes `systemctl is-active firewalld/ufw`). No installer-level firewall rule is added — the in-app helper covers all platforms.

- **Everything runs on the GUI thread, event-driven — no worker thread, no asyncio, no locks.** The only two things touched (grabbing the player view and mutating the scene) both have GUI-thread affinity, so keeping the sockets on the GUI thread removes all cross-thread marshalling. (Contrast the *transcode* path, which threads only because it shells out to an ffmpeg **subprocess**.)
- **Two servers.** A `QTcpServer` serves one self-contained HTML/JS page (the `INDEX_HTML` string, `__WS_PORT__` substituted) on `GET /` at `http_port`. A `QWebSocketServer` on `ws_port` carries the rest: server→client **binary JPEG frames** (~12.5 fps, `FRAME_INTERVAL_MS`) + **token-state JSON**; client→server **move commands**. The served page injects `ws_port`, so the two ports need not stay adjacent.
- **Ports: uncommon default + auto-fallback + optional custom.** The default base is **`DEFAULT_WEB_PORT = 47800`** (ws `47801`) — deliberately high/uncommon so it rarely collides with dev servers on 8080/8081. `WebServer.start()` **probes upward** for a free port (`_first_free_port`/`_first_free_ws_port`, span `PORT_SCAN_SPAN=40`) instead of hard-failing on a conflict, and the URL/QR always reflect the **actually-bound** `http_port`. `MainWindow._start_web_sharing()` re-applies the configured base (`_web_base_port()`) on every start (so a dialog change or a prior fallback value is reset first). The base is the app default unless the user ticks **"Use a custom port"** in the dialog (`web_port_custom` + `web_port`, both persisted in `settings.json` as `webPortCustom`/`webPort`); the port controls are editable only while stopped. A GUI-thread `QTimer` (`_tick`) grabs `player_window.canvas_view.viewport().grab()` → JPEG (`MAX_FRAME_WIDTH`/`JPEG_QUALITY`) and broadcasts. **Requires the Player Window open *and visible*** — `_player_canvas()` returns None (→ `view:false`, no frame) when there's no player window **or it's hidden** (disabling the player view *hides*, not deletes, the window; the `isVisible()` check is what makes the client's "Waiting for the GM…" message reappear and clear the last frame instead of freezing on it). `lan_ip()` uses the UDP-connect trick (works offline) with a `QNetworkInterface` fallback.
- **The web client (`INDEX_HTML`) supports pan / zoom / rotate.** Frame + token-overlay live in a `#view` layer with a `translate()·scale()·rotate()` transform; one-finger drag pans, pinch / mouse-wheel zooms (around the gesture point), the **⟳ button rotates the map 90°** (each player can orient the table toward themselves — purely client-side, doesn't affect the GM or other clients), double-tap resets to fit. `fitView()` centres the *rotated* bounding box; token drag converts pointer→canvas with an explicit **inverse transform `screenToCanvas()`** (undoes translate→scale→rotate) rather than `getBoundingClientRect`, so dragging stays correct at any rotation. Token handles `stopPropagation` on pointerdown so dragging a token never pans. On `view:false` the client clears the canvas and ignores stale frames.
- **The pixel stream gives rich rendering for free** (maps/fog/video/alpha all just work because it's pixels); the state-sync layer only ships **token rects** so the client can overlay draggable handles on top. `_state_json()` lists **only tokens that are `is_token` AND `player_controllable` AND `visible_to_player`**, each as a **viewport-normalised** rect (`cv.mapFromScene(...) / viewport size`) — normalising over the whole grabbed viewport is **letterbox-proof** and means the browser does no scene math. Hidden/NPC tokens never expose a handle.
- **Single mutation chokepoint:** `MainWindow.apply_token_move(token_id, nx, ny)` (called from the WS message handler, GUI thread). It maps the normalised **centre** back through `cv.mapToScene`, finds the matching token **and re-checks `player_controllable`** (server-side authority — a forged id for a non-permitted token is ignored), `setPos`es it, then reuses the existing **grid-snap (`itemChange`) + non-overlap resolve (`_resolve_token_overlap`)**. The shared scene means GM + Player views update automatically; the next frame tick reflects it. The web client predicts the drag locally and ignores server state for the token it's actively dragging until release (no snap-back fight).
- **Stable per-item ids:** `MainWindow._token_id(item)` assigns a lazy `uuid` hex (`item.token_id`), persisted as `id` in the map JSON; **`player_controllable`** (per-token, toggled from the item context menu's "Allow/Disallow Player Control", multi-select via `_token_targets`) saved as `playerControllable`. `open_map` restores both (generating an id if missing); **paste assigns a fresh id** (a pasted token is a new instance) but carries the flag. These are the "stable IDs + apply-mutation seam" prerequisites the future-direction notes call for — now real.
- **Two token rings, painted in this order in `InteractivePixmapItem.paint`:** first the **solid gold** "party member" ring (`items._paint_party_ring`, gated `is_token and self.in_party`) — a character-identity marker drawn in **both** the GM and player views; then, **over it**, the **green dashed** "player-controllable" ring (`items._paint_player_control_ring`, gated `is_token and player_controllable and not in_player`, GM-only). The green ring is drawn at the **same ellipse/width** as the gold one with a **50/50 dash** (`setDashPattern([2,2])`), so half the gold shows between the green dashes (= "this party token is web-draggable"). Both are scene-px so they survive zoom-out. The `in_party` flag is kept in sync by **`MainWindow._refresh_party_token_rings()`** (membership is by *asset*, so every on-map token sharing a roster asset gets the gold ring); see Player party.
- **Player tokens render *through* fog.** Fog is painted in `Canvas.drawForeground` (above every item regardless of z), so to keep player tokens visible over it, `drawForeground` **re-paints each `in_party` token on top of the fog fill** — iterating `QGraphicsView.scene(self).items()` (the real attached scene, since the player canvas's `self.scene` attribute is its own empty one) and calling `it.paint(painter, opt, self.viewport())` under `it.sceneTransform()` (the viewport widget makes paint() pick the right per-view chrome and honour `visible_to_player`). Only happens while `fog_enabled`.
- **Lifecycle:** `WebServer` is created lazily on first start and stopped in `MainWindow.closeEvent`. Auth is intentionally open on the LAN for the in-person case (add a room code later). One small pip dependency: **`qrcode`** (pure-Python; only for the join QR — `webserver.make_qr_pixmap` renders the matrix to a `QPixmap` by hand, so **no Pillow**, and a missing package degrades to a URL-only dialog). `QtNetwork`/`QtWebSockets`/`QtWidgets` all ship with PySide6.

### Player party (one saved roster of player tokens)
A **party** is a single roster of player tokens that can be stamped onto any map, stored in `settings.json` (not per-map). It's a thin layer over tokens — a member is just a *reference* to a library token plus captured appearance.

- **State / persistence:** `MainWindow.party_members` is a list of `{asset, name, color, controllable}` dicts (`asset` = library-relative token path = the identity key; `color` = captured `token_color_override`, may be null; `controllable` = captured `player_controllable`; `name` = filename stem, display only). Loaded in `_load_settings` and written in `closeEvent` under a top-level **`party`** key (`{"members": [...]}`) — the only non-window-geometry data in `settings.json`. No image data is duplicated: the token PNG + `.tokenmeta` already live in the library.
- **Membership** is edited from the **token right-click menu** (`items.py`, tokens only): a checkable **"Add to Party"** and a checkable **"Allow Player Control"** (both multi-select via `_token_targets`; control also `item.update()`s to toggle the GM ring). Add captures the token's current color + control flag (re-adding refreshes them); the two checkboxes are **independent** (adding to the party does not force control). Routed via `view.window().set_party_membership(items, bool)` / `is_in_party(item)`.
- **The party controls live in the `.ui`** (added in Qt Designer, regenerated into `ui_mainwindow.py`), wired in **`_wire_party_controls()`** (called from `__init__` after `_load_settings`). Expected object names — all optional, `getattr`-guarded so the app still runs if absent: **`placeParty_btn`** (QPushButton → `place_party`), **`removeParty_btn`** (→ `remove_party_from_map`), **`clearParty_btn`** (→ `clear_party`), **`partyMembers_comboBox`** (QComboBox, live roster display). **`_refresh_party_ui()`** repopulates the combo (token thumbnail + name, or "(empty)") and enables/disables the buttons (Place/Remove need a map open; Clear needs a non-empty roster); call it after any roster change — already hooked in `set_party_membership`, `clear_party`, and `_update_map_ui_state` (so map open/close updates the buttons).
- **`place_party()`** stamps every member **not already on the current map** (dedup by `asset_filename` via `_assets_on_map()`) at the GM-view centre, each via `_place_asset` (which now **returns the placed item**, not a bool — backward-compatible truthiness for `Canvas.dropEvent`), then sets `player_controllable` + re-bakes the ring color (`_bake_token_with_color`). Sequential placement → each fans out through `_resolve_token_overlap`. Existing party tokens are left untouched; missing library files are skipped with a warning. **`remove_party_from_map()`** deletes on-map tokens whose asset is in the roster (roster intact, and they stop travelling); **`clear_party()`** (the "Disband" button) removes the player tokens from the map (`remove_party_from_map`) **and** empties the roster.
- **Player tokens are a session overlay: NOT saved with maps, and they travel.** A token whose asset is in the roster is a "player token." `_save_to_path` **skips** them (so map JSON never contains player tokens). On a map switch they carry over: `open_map` calls `_snapshot_player_tokens()` *before* `scene.clear()`, then `_restore_player_tokens(snaps)` *after* the new map's items load (before `restack_layers`) — re-creating each at its prior position, then `_resolve_token_overlap` bumps any that land on the new map's tokens (positions otherwise kept; `token_id`/colour/`player_controllable`/`in_party` preserved; an asset already present is skipped to dedup legacy-saved party tokens). **`new_map` does NOT carry them** (a fresh map starts empty — re-add with Place Party); only `open_map` travels. (Legacy maps that still have player tokens saved self-heal: the next save drops them.)
- **Gold ring marker.** A placed token whose asset is in the roster shows a **gold ring** in both views (see Per-item player visibility → token rings; player tokens also render *through* fog). `_refresh_party_token_rings()` re-syncs the per-item `in_party` flag and is called after every roster change (`set_party_membership`, `clear_party`), after `place_party`, after a token `_place_asset`, and at the end of `open_map`. **`_rewrite_asset_refs` (asset rename/move) also remaps the roster's `asset`/`name`** (membership is by path) and re-syncs, so a renamed/moved party token keeps its membership + gold ring and Place/dedup stay correct. It's *not* persisted — `in_party` is derived from the (settings-scoped) roster each session, so it follows the party across maps for free.

### Fog of war
Fog is a per-scene **`QPainterPath` in scene coordinates** (`Canvas.fog_reveal_path`) — the union of every revealed brush stamp. `Canvas.drawForeground` fills the visible rect **minus** that path with `Canvas.fog_color` (GM at `gmfog_opacity`, player at 1.0). Key points:

- **Resolution-independent, tiny, world-anchored.** Because the path is in scene coords (which don't shift when the canvas grows/shrinks), reveals stay locked to the map for free — there is no mask sizing, no `fog_origin`, no realignment. An empty path = fully fogged. `MainWindow._reset_fog_mask(hidden=True)` resets to fully fogged (`hidden=False` = whole scene revealed).
- **Reveal/hide keep the live stroke as a separate, simplified path (perf-critical).** During a drag, `reveal_at`/`hide_at` union each stamp (`Canvas._brush_stamp`) into a scratch `_fog_stroke` path and re-`simplified()` it — this is cheap and *bounded* because the stroke holds only the current drag's outline, never the whole revealed history; the big committed `fog_reveal_path` is untouched until release. That's the fix for the brush lagging the cursor (the old per-stamp `united()` grew the committed path → O(n²) over a stroke). On **stroke end** (`mouseReleaseEvent`) `Canvas.commit_fog_stroke()` folds the stroke into `fog_reveal_path` with one `united()`/`subtracted()` + `simplified()`. `simplify_fog()` (called before save) commits any pending stroke first.
- **The circular brush is a polygon, not `addEllipse` (perf-critical).** `_brush_stamp` approximates a circle with a straight-segment polygon (vertex count scales with radius). A bezier ellipse is ~13 path elements, so unioning many balloons the merged fog path to tens of thousands of curve elements and every boolean op + repaint slows down (and worsens as more is revealed); polygonal stamps keep it ~5–10× smaller and rendering ~17× faster (≈1.3 ms vs ≈22 ms/frame in a heavily-revealed scene), at no visible cost for soft fog. Square brush is already `addRect`.
- **`drawForeground` never uses Clear-compositing.** It fills `rect − fog_reveal_path` (then, for an active stroke, `− _fog_stroke` for reveal or `∪ _fog_stroke` for hide) in a single `fillPath` at the view's opacity. Erase-style `CompositionMode_Clear` would zero the *opaque* viewport backing store to **black** instead of revealing the scene beneath, so the stroke is always folded into the fogged region before filling. The Hide tool creates holes; odd-even fill renders and round-trips them correctly. The player view mirrors `_fog_stroke`/`_fog_stroke_mode` via `_push_fog_to_player()` so it shows the dab in progress.
- **`drawForeground` uses the scene-coord path directly, never `self.scene`.** This sidesteps a latent gotcha: `Canvas.__init__` assigns `self.scene = QGraphicsScene(...)`, which *shadows* the `QGraphicsView.scene()` method. The `PlayerWindow`'s canvas gets the shared scene via `setScene(...)` (a view method), but its `self.scene` **attribute** still points at its own empty scene. Since the painter is already in scene coords and we consult only `fog_reveal_path`, the same code is correct for both windows.
- **Player sync copies the path, not a shared object.** `reveal_at`/`hide_at` replace `fog_reveal_path` with a *new* object each stroke, so the player's reference goes stale — `MainWindow._push_fog_to_player()` (hot path, per stroke) and `sync_fog_to_player_view()` (structural) re-point it and set `fog_enabled`.
- **Serialization is inline in the map JSON.** `fog_path_to_json()` flattens the path to a list of closed polygons via `toSubpathPolygons()` (curves from circular stamps become line segments — lossless for hard-edged fog, keeps JSON small); `fog_path_from_json()` rebuilds it with `moveTo`/`lineTo`/`closeSubpath`. Stored under the `fogReveal` key; **no sibling PNG**. There is **no backward compatibility** — maps without `fogReveal` open fully fogged; legacy `fogMaskGM`/`fogOrigin`/`_fog.png` are ignored (the delete/rename/move handlers still relocate any lingering `_fog.png` alongside its map, harmlessly).
- **`main()` calls `QImageReader.setAllocationLimit(0)`** — now only to allow large *map images* to decode past Qt's default 256 MB limit (fog no longer loads an image at all).
- **Startup splash (`SplashScreen`).** `main()` shows a frameless `SplashScreen` (enlarged app `ICON_PATH` + a " Arcane Atlas" title + a `QProgressBar`), calls **`ensure_visible()`** (pumps the event loop until the window is actually mapped+painted — `show()` only *queues* the first paint, so without this the splash flashed at the very end instead of being visible during startup), then: `advance_to(45)` → build `MainWindow` (~instant) → `advance_to(80)` → `win._ensure_preview_player()` (the ~2s FFmpeg load — bar **holds at 80**, splash stays painted) → `advance_to(96)` → `win.show()` → `advance_to(100)` → close. `advance_to(target, pace_ms)` tweens visibly (1% steps, `processEvents` + `QThread.msleep`) and never goes backwards.
- **Why the bar can't animate *during* the FFmpeg load (don't re-attempt the "thread" fix).** The ~2s cost is the **process-global FFmpeg plugin load**, triggered by the first multimedia object. It can't be smoothly animated single-process: on the **main thread** it blocks the thread; on a **worker thread** (a `QThread` constructing a throwaway `QMediaPlayer`) it **holds the Python GIL** for the whole load, starving the main thread's animation timer (measured: ~2 timer ticks in 2.1s vs ~84 if smooth). So a worker thread buys nothing for animation and adds risk (multimedia on a worker, on the fragile PySide6/shiboken stack) — it was tried and reverted. The only way to truly animate through it is a **separate process** for the splash (heavier; not done). Hence the bar just holds at 80 during the load while the splash stays visible.

### Asset categories & the import dialog (backgrounds / objects / tokens)
The library has three category subfolders (`ASSET_CATEGORIES = ("backgrounds", "objects", "tokens")`). **Category is asked, not assumed.** When an **external** file is dropped on the canvas, `_place_asset` calls `_ask_asset_category()` — a modal radio-button `QDialog` (Background / Object / Token) whose **last choice is remembered** in `self._last_asset_category` and preset for the next import; cancel aborts the drop. **Library** assets (dragged from the Assets browser, or reconstructed when a map loads) keep their existing folder (`_library_category_of()`), so they're never re-prompted and never re-imported (`_asset_is_in_library()`).

Category drives how an asset is **rendered and transcoded** — this is the important part:

- **Backgrounds** (and any opaque video) → the **video pipeline**. `InteractivePixmapItem` for images, `InteractiveVideoItem` (`QGraphicsVideoItem` + `QMediaPlayer`) for video. Non-hardware codecs are transcoded to **H.264/mp4** (see below). Transparency is *not* preserved (the video backend can't — see Fog/alpha note below), which is fine for full-screen battle maps.
- **Objects** → may be **transparent animations**. If a dropped *object* video carries alpha (`video_has_alpha()`), it's converted to an **animated WebP** and rendered by `AnimatedItem` via the **image pipeline** (`QMovie`), which *does* composite per-pixel alpha. An opaque object video falls back to the normal H.264 video path. Object *images* (incl. transparent PNGs) are just imported into `objects/`.
- **Tokens** → **images only** (rejected for video/anim with an info dialog). An external image opens the **tokenizer** (see below) which bakes a round, transparent-cornered token PNG into `tokens/`; the placed item is a fixed-size `InteractivePixmapItem` that **can't be resized on the map**. A token *already in the library* (dragged from the Assets browser, or reconstructed on load) is placed directly — no tokenizer re-run.

The item type is chosen from the **final asset extension**, not the category guess (`.webp` → `AnimatedItem`, else video → `InteractiveVideoItem`), so a failed/declined transparency conversion that falls back to a plain video still produces the right item. Serialized item `type`s are `"image"`, `"video"`, `"anim"` (see Persistence) — a token serializes as a plain `"image"` (its token-ness is re-derived from the `tokens/` category on load).

### Tokens & the tokenizer (round VTT-style tokens)
A **token is a baked round PNG** in the `tokens/` library, rendered by an ordinary `InteractivePixmapItem` (the category chosen in the import dialog). There is **no TokenItem class** — "token" is a *mode* of the pixmap item, flagged by `InteractivePixmapItem.is_token`. The whole feature is image-pipeline only (no video/alpha-WebP concerns).

- **The tokenizer** (`arcaneatlas/tokenizer.py`, `TokenizerDialog`) is a self-contained modal — it imports only PySide6, **never `main`** (same one-way rule as `items`/`transcode`). It takes a source `QImage`, lets the user **pan (drag) and zoom (scroll/slider)** the image behind a fixed circular crop, pick a **size** (1×1 / 2×2 / 3×3 / 4×4 inch → a `sizeInches×72`px circle), and a **round border ring** — color from **8 fixed swatches** (`BORDER_COLORS`, a checkable exclusive `QButtonGroup`). **Every token has a ring** (the on/off checkbox was removed — `border_enabled` is always `True`, and re-editing a legacy ring-less token forces it back on); the ring is **always thin** (`BORDER_WIDTH = 0.04` of the diameter — the thickness picker was removed). `result_params()` always serializes `borderEnabled: True` + `borderColor` (hex) / `borderWidth` (always `0.04`); re-edit snaps a restored colour to the nearest swatch (`_select_color`) and forces the width back to `BORDER_WIDTH`. The actual bake lives in a **module-level `bake_token(src, params)`** (not a dialog method) so it can be reused without a dialog — `result_image()` is now just `bake_token(self._src, self.result_params())`, and the per-instance recolor (below) calls it too. It produces a square `ARGB32` PNG (transparent corners, content clipped to the circle, border stroked just inside the edge); `result_params()` returns the crop in **resolution-independent source coords** — `cx`/`cy` (source point at the circle centre) and `fitDiameter` (source px spanning the circle) — so a re-bake at a different size stays correct. `GRID = 72` mirrors the canvas inch.
- **Storage = baked PNG + sidecar.** `MainWindow._build_token_item()` runs the tokenizer on an external image, writes `tokens/<name>.png` (the reusable library asset, collision-named via `_unique_token_png`) **and** a sibling **`tokens/<name>.tokenmeta`** JSON sidecar holding the tokenizer params + the (resolution-capped, `_prep_token_source`) source image as base64. The sidecar makes the token **self-contained**: re-edit and library-reuse both work without the map JSON storing any token-specific fields. Sidecars use a non-`ASSET_EXTS` extension, so they're **hidden in the Assets browser** (which sets `setNameFilterDisables(False)`).
- **Re-edit:** there are two entry points, both funnelling into **`_edit_token_file(png_path)`** (reads the sidecar params+source, re-opens the tokenizer, re-bakes **in place** overwriting the same PNG + sidecar, then refreshes every placed item referencing that token and the Assets-tab preview). (1) Each token item's context menu gains **"Edit Token…"** (only when `is_token`), routed via `view.window().edit_token(item)` → `_edit_token_file`. (2) The **Assets browser** (`on_asset_context_menu`) shows **"Edit Token…"** on any file under the `tokens/` category. If the sidecar is missing (legacy/lost), it falls back to using the baked PNG as the source — size/border still editable, but the crop can't be widened.
- **No resize on the map:** `InteractivePixmapItem._handle_positions()` returns `{}` when `is_token`, so there are no resize handles (the item is still movable/rotatable). Size is changed only by re-editing in the tokenizer.
- **Always grid-snapped:** tokens snap to the canvas grid on every move/placement/load. `InteractivePixmapItem.__init__` adds `ItemSendsGeometryChanges` and `itemChange()` rounds the proposed position to a `grid_size` (72px) multiple **when `is_token`** (non-tokens pass through unchanged). The token pixmap is a whole number of inches square (`N×72`px) and grid lines fall on multiples of `grid_size` in scene coords, so snapping the top-left origin lands the token exactly on `N` cells. This fires for programmatic `setPos` too (placement in `_place_asset`, restore in `open_map`), so legacy off-grid tokens re-snap on load.
- **Tokens never overlap each other.** `MainWindow._resolve_token_overlap(item)` bumps a token to the nearest free grid cell, called after a **drop** (`InteractivePixmapItem.mouseReleaseEvent`, tokens only), a **placement** (`_place_asset`), and **each pasted token** (`_paste_internal_items`, so a multi-token paste fans out instead of stacking). **Multi-selection moves:** a group drags together but only the grabbed item gets the release event, so `mouseReleaseEvent` resolves **every selected token** (grabbed one first, to anchor it where dropped when that cell is free; sequential resolution converges to mutually non-overlapping cells since each call tests against all other tokens' current positions). It compares **scene bounding rects** (rotation-safe) of the other tokens only — tokens may still sit on backgrounds/objects — and spirals outward ring-by-ring in whole `grid` steps (Chebyshev distance; nearest candidate wins) via `moveBy`, preserving the grid snap. Rects are shrunk 1px before the `intersects` test because `QGraphicsPixmapItem.boundingRect()` carries a 0.5px-per-side margin (so a cell-filling token is ~73px) — without the shrink, tokens in *adjacent* cells would read as overlapping and get pushed a full cell apart. Resolution happens at **rest** (on drop, not live during the drag).
- **Per-instance border colour (not the shared asset):** a placed token's context menu has a **"Change Token Color"** submenu of the 8 `BORDER_COLORS` swatches → `view.window().recolor_tokens(_token_targets(self), hex)`. `_token_targets(item)` returns every selected token when the clicked one is part of a multi-selection, else just it — so you can recolour several at once. `MainWindow._bake_token_with_color(item, hex)` reads the token's sidecar (params + source) and `bake_token`s it with `borderEnabled=True`/`borderColor=hex`, then `setPixmap`s the result on **that item only** — the library `tokens/<name>.png` is **never rewritten** (verified byte-identical). The override is stored as `item.token_color_override` (hex, or `None` = use the library PNG as-is) and persisted per-item as **`tokenColor`** in the map JSON; `open_map` and clipboard paste re-bake it, and `_edit_token_file` re-applies each item's override on top of a freshly-edited base. Contrast with **Edit Token…**, which overwrites the shared PNG for *all* placements.
- **Persistence:** a token saves as a normal `"image"` item referencing `tokens/<name>.png` (plus the optional per-instance `tokenColor`, above). `open_map` sets `it.is_token = (asset_category == "tokens")` for image items, restoring the resize lock + Edit Token entry. Because params live in the sidecar (next to the asset, not in the map), a token dragged into a *second* map re-edits with full fidelity too.
- **Library file management carries the sidecar.** `_relocate_asset` moves/renames a token's `.tokenmeta` alongside its PNG (folder moves take contents automatically, so only the file case is handled); `_delete_asset` removes it. Cross-category moves are still rejected (tokens stay in `tokens/`).
- **Reuse caveat:** multiple placed items can reference the same `tokens/<name>.png`; editing **overwrites the shared PNG**. `_edit_token_file` refreshes every matching item in the *currently open* map, but placements in *other* (closed) maps update only on reload.

### Text boxes (map annotations — the one non-asset-backed item)
`TextBoxItem` (`items.py`, a `QGraphicsObject`) is a free-standing text annotation — **the only map item with no library asset**. Its text + styling live **inline in the map JSON** as `type:"text"` (no `asset`), and it serializes itself: `to_json()` / `apply_json(d)` on the item (called by save/load, copy/paste, and the signature). It never imports `main` — it reaches the window via `view.window()` like the other items.

- **Create it by right-clicking empty canvas → "Add Text Box".** Right-click is *pan*, and **on this platform the context-menu event fires on button *press***, so the menu is **NOT** driven from `contextMenuEvent` — showing a blocking menu (or clearing `_panning`) on press would kill right-drag panning before the drag starts (this was a real regression; don't reintroduce a `Canvas.contextMenuEvent` override). Instead `Canvas.mouseReleaseEvent` calls **`_maybe_show_canvas_menu(vpos, gpos)`** only when the right button was released **without dragging** (`_pan_moved` stays False — set in `mouseMoveEvent`), and *after* clearing `_panning` (so a menu/dialog shown from it can't leave the view stuck panning). It skips when a **selected** map item is under the cursor (that item shows its own menu via the default context-menu path) — so the menu still appears over an *unselected* background, using `self.items(pos)` (z-stack, not `itemAt`, so the full-view `PlayerViewItem`/grid don't mask it). GM-view only (`is_player_view` guard). `MainWindow.add_text_box(scene_pos)` creates the box centred on the click, opens the editor, and discards it if the brand-new box's editor is cancelled (auto-creates a map first if none is open). The same menu also carries **"Import File…"** → `MainWindow.import_file(scene_pos)` (a `QFileDialog` filtered to `ASSET_EXTS`, then `_place_asset` — identical to a drag-drop import, same background/object/token category prompt). The menu is **gated on Lock Assets being off** (text boxes + imports are 'objects'). **Item context menus (`InteractivePixmapItem.contextMenuEvent` etc.) keep clearing `view._panning` themselves** — that's what prevents *their* on-press menu from sticking the pan.
- **Edit by double-click or context-menu "Edit Text…"** → `MainWindow.edit_textbox(item)` → **`TextBoxEditDialog`** (in `main.py`, near `Canvas`). Rich styling: multi-line text, font size, **bold/italic**, **alignment** (left/center/right), and **text / background / border colours** via `ColorPickerButton` (alpha-capable, so the background can be translucent or off), plus border width. The dialog **live-previews** onto the item as you edit; `edit_textbox` snapshots `to_json()` first and **reverts on Cancel**.
- **Interaction:** movable; **resize = drag the right-edge handles** (`tr`/`br`) to set the **wrap width** (height auto-fits the text — `boundingRect` measures wrapped text via `QFontMetrics`). Resize reads `event.pos()` (item coords) so it's **rotation-proof**. **Rotate 90°** via the context menu (`setRotation` about the box centre). Same **hide-from-players** marker + GM-only selection chrome as the other items (paint gates on `_painting_in_player_view`).
- **Layer:** class attr `asset_category = "objects"`, so `_item_layer` slots it into the **objects** z-band and the Layers "Objects" group with no special-casing; `_item_display_name` shows a quoted text snippet. It's included in `_map_items()`, the canvas extent (`_content_rect`), the saveable signature (so editing text counts as an unsaved change), and copy/paste (`_media_item_types`, `_clip_entry_for_item`/`_item_from_clip_entry` — text has a `width`, not a `size` pair, so the paste bbox helper `_wh` handles both). The asset-copy loop and `_save_to_path`'s asset branch skip it (no `asset_path`/`asset_filename`).
- **Adding a fourth media item type?** Keep the four `isinstance(... TextBoxItem)` sites in sync: `_map_items`, `_content_rect`, `_saveable_signature`, and `_toggle_player_visibility`'s media tuple — plus the save/load/clip branches keyed on `type == "text"`.

### Why transparency uses a separate pipeline (alpha is the whole reason WebP exists here)
**Qt's FFmpeg video backend (`QMediaPlayer`/`QGraphicsVideoItem`) drops the alpha channel of WebM (VP8/VP9) videos** — it decodes to YUV420P/NV12 and paints an opaque rectangle (verified: even a pure-software VP8 decode loses alpha; the native ffmpeg VP8/VP9 decoders don't surface the WebM alpha side-channel, only `libvpx` does). H.264 has no usable alpha at all. The **only** pipeline in this Qt build that composites per-pixel alpha is the **image** pipeline (`QMovie`/`QImageReader`). Hence: transparent animated objects are transcoded to animated WebP and rendered by `AnimatedItem`, never by the video player.

### Video transcoding on import (codec conversion)
Videos whose codec the GPU can't decode in hardware (notably **VP8**) fall back to expensive CPU decoding when several play at once. So non-hardware codecs are transcoded on import. All transcode helpers live in **`transcode.py`**.

- `HW_VIDEO_CODECS = {h264, hevc, vp9, av1}` are left as-is; anything else is transcoded. `probe_video_codec()` (ffprobe) picks the path; `_run_ffmpeg()` is the shared runner (progress + cancel); `_ffmpeg_tool()` locates the binary.
- **H.264 path** (`transcode_to_h264`, opaque/background video): `libx264`/`yuv420p` mp4. The `_import_video()` prompt asks convert-or-keep; on keep/cancel/failure it plain-copies the original (software decode) so a drop never breaks.
- **Animated-WebP path** (`transcode_to_animated_webp`, transparent objects): **must use `-c:v libwebp_anim`, NOT `libwebp`** — plain `libwebp` blends each frame onto the previous canvas, so a moving/fading animation leaves a *trail of every prior frame*; `libwebp_anim` does per-frame disposal. **VP8/VP9 sources must be decoded with `libvpx`/`libvpx-vp9`** (`pre` args) — ffmpeg's native decoders drop the alpha side-channel, so the WebP would come out opaque. Output is `yuva420p`, `-loop 0`.
- **Conversions run on a worker thread.** `MainWindow._run_transcode()` runs the blocking ffmpeg on a `threading.Thread` behind a window-modal `QProgressDialog` (with **Cancel**), pumping a nested `QEventLoop` so the GUI stays responsive (same pattern as `QMessageBox.exec()`). `_TranscodeSignals` (a `QObject`) marshals progress/done from the worker back to the GUI thread (queued). `cancel_cb` polling in `_run_ffmpeg` kills ffmpeg and removes the partial file. Returns `(ok, cancelled)`.
- **Progress is a pulse for WebP.** The animated-WebP muxer buffers the whole clip and reports progress only **once, at the very end** — there's no real % to show, so `_run_transcode` animates a left↔right **pulse** on a GUI-thread `QTimer` (the busy-bar style rendered static here). It switches to a true determinate bar only when genuine intermediate progress arrives (H.264 on a longer video). Read the cancel flag **before** `dlg.close()` — `QProgressDialog.close()` itself emits `canceled()`.
- **Existing library:** there used to be a "Optimize Library for Hardware…" batch-convert command (removed). A transparent object video that lives in the library as `.webm` will render as an opaque box if dragged from the Assets browser; transparent assets should be pre-converted to `.webp` (or re-dropped as an external Object). (`_update_map_refs` — the disk-only exact-match ref rewriter that command used — is now unused dead code.)
- **ffmpeg is bundled, not assumed.** `_ffmpeg_tool()` resolves `resources/bin/ffmpeg(.exe)`/`ffprobe(.exe)` first (shipped via the `Tree("arcaneatlas/resources")` in the spec), then the system PATH. **For Windows release builds drop static LGPL `ffmpeg.exe`/`ffprobe.exe` into `arcaneatlas/resources/bin/`** (git-ignored; see the README there). The build must include `libvpx` and `libwebp_anim` support or the transparency path silently produces opaque/trailing output. On Linux/dev the system ffmpeg is used.
- Subprocess calls pass `creationflags=_SUBPROCESS_FLAGS` (`CREATE_NO_WINDOW` on Windows) so the windowed build doesn't flash a console.

### Assets browser (Assets + Tokens tabs)
- `MainWindow._build_assets_tab()` builds a `QTreeView` over `asset_dir` (categories `backgrounds/objects/tokens` shown as folders) backed by **`AssetFileModel`** (a `QFileSystemModel` subclass) filtered to `ASSET_EXTS`. The user can **create subfolders, rename, move, and delete** within the categories (see below); the three category roots themselves are protected (`_is_category_root` — can't be renamed/moved/deleted, only added to).
- **Tokens live in their own tab.** `_build_tokens_tab()` adds a second `QTreeView` (`token_view`) over the **same `AssetFileModel`** but rooted at `tokens/`, with an image-only preview (`token_preview_label` — tokens are always static PNGs, so no video backend, unlike the Assets tab's `QStackedWidget`/`QVideoWidget`). The Assets tab **hides its `tokens/` row** so the two don't overlap. Both happen in **`_on_asset_dir_loaded(path)`** (wired to `AssetFileModel.directoryLoaded`, since `QFileSystemModel` populates async): when `asset_dir` finishes loading it `setRowHidden(tokens_row, …, True)` on `asset_view` and `setRootIndex(tokens_idx)` on `token_view`. Both views share `on_asset_context_menu` / `_on_asset_double_clicked` and one model — `on_asset_context_menu(point, view)` takes the originating view (connected via a `lambda`) so "Edit Token…", drag-to-canvas, rename/move/delete all work from either tab.
- `ASSET_EXTS` includes `*.webp` (animated transparent objects) alongside the image/video extensions.
- **Organising assets & keeping map refs intact.** Asset refs in map JSON are library-relative POSIX paths (`backgrounds/x.png`), so moving/renaming a file would break every map that uses it. Because the move happens *inside* the app, we always know old→new and rewrite refs everywhere:
  - **Drag-move** within the tree goes through `AssetFileModel.dropMimeData` → `MainWindow._move_assets_into` (the view is `DragDrop`, `defaultDropAction = Move`). **Context menu** (`on_asset_context_menu`) offers New Folder / Duplicate / Rename / Move to… / Delete. **Duplicate** (`_duplicate_asset`, files only) copies the asset to a collision-free "`<name> copy`" file — and for a token also copies the `.tokenmeta` sidecar, so the copy is a fully independent, re-editable token; no map refs need rewriting (it's a brand-new unreferenced asset). Right-clicking **empty space** in a browser targets the view's **root dir** (so the Tokens tab's `tokens/` root still offers "New Folder…"); the library root itself offers nothing (categories are fixed). **F2** renames the selected item in whichever browser has focus (per-view `QShortcut`, `WidgetShortcut` context → `map_view.edit()` for maps, `_rename_asset` for assets/tokens; category roots are skipped). Rename and move both funnel into `_relocate_asset(src, dst)` (does the `shutil.move` + ref rewrite; rejects collisions, folder-into-itself, and **cross-category moves** — an asset can only move within its own top-level category, enforced by comparing `_library_category_of(src)` vs `_library_category_of(dst)`). **`_rename_asset` re-appends the original extension** if the typed name omits it — the browser filters to `ASSET_EXTS` with `setNameFilterDisables(False)` (hides non-matching files), so a token renamed to "goblin" (dropping `.png`) would otherwise silently vanish from the list.
  - `_rewrite_asset_refs(old_rel, new_rel, is_dir)` rewrites **every map JSON on disk** *and* the **open map's live scene items** (`item.asset_filename`/`asset_path`, so a later save writes the new ref instead of re-importing from the moved path). `_remap_ref`/`_ref_matches` handle both exact-file and folder-prefix matches (the boundary check avoids `dun` matching `dungeon`). (`_update_map_refs` is an older disk-only exact-match version, now unused after the Optimize-Library command was removed.)
  - **Delete can't be auto-repaired**, so `_delete_asset` first calls `_maps_referencing()` and warns with the list of affected maps before deleting (user can proceed; those items then show as missing on next load).
  - **Load-time safety net:** `open_map` **skips any item whose asset file is gone** (collecting them into a "Missing Assets" warning) instead of spawning a broken pixmap/black video box — this covers OS-file-manager moves the app can't intercept. **Never fall back to basename lookup** (two categories can both hold `goblin.png`).
  - **Drag-to-canvas must stay a Copy, not a Move.** With the tree defaulting to `MoveAction`, an accepted Move on an external target makes Qt's `clearOrRemove` **delete the dragged library file**. So `AssetFileModel.supportedDropActions` advertises `Copy | Move`, and `Canvas.dragEnter/dragMove/dropEvent` force `event.setDropAction(Qt.CopyAction)`. If you touch the canvas drop handlers, keep that — accepting the proposed Move there silently deletes assets.
- **Adding to a map:** drag to the canvas (the model's `text/uri-list` drag is handled by `Canvas.dropEvent` → `_place_asset`) or **double-click** (`_on_asset_double_clicked` → `_place_asset` at the GM view center). Library files are detected as already-imported by `_import_asset`/`_import_video`, so adding never duplicates (and library drags skip the category dialog).
- **Preview pane:** a `QStackedWidget` at the bottom of the tab — index 0 a `QLabel` (scaled image thumbnail, **animated WebP via `QMovie`**, or placeholder text), index 1 a `QVideoWidget`. `_on_asset_selected` stops the shared `asset_preview_player` (a looping `QMediaPlayer`) *and* `_stop_preview_movie()` before swapping; `.webp` previews animate in the label (image pipeline), regular videos play looping/silent in the `QVideoWidget`.
- **No `QAudioOutput` (preview is silent).** `_ensure_preview_player` deliberately does **not** create/attach a `QAudioOutput` — the preview has no sound regardless, and constructing/muting one opens the system audio backend (**pipewire**), which **intermittently segfaults on a cold start** (`libpipewire-module-protocol-native`, crashing in `_ensure_preview_player` during the eager startup warm). A player with no audio output just plays video silently, matching `InteractiveVideoItem` (which also sets only a video output). `asset_preview_audio` stays `None`. **Don't reintroduce a `QAudioOutput`** — if previews ever need sound, expect to deal with that pipewire crash.
- **The video preview widgets are built by `_ensure_preview_player()`** (`asset_preview_video`/`_player`/`_audio` start as `None`, index 1 of the stack added on build). The method is **lazy + idempotent** — anything touching `asset_preview_player` must still null-check it, since it can be `None` until the first build. Constructing `QVideoWidget` initialises the Qt FFmpeg multimedia backend (~2s).
- **The backend is warmed eagerly at startup, on the main thread, behind the splash** — `main()` calls `win._ensure_preview_player()` after building `MainWindow`, paying the ~2s FFmpeg-plugin-load cost up front (the splash bar holds at 80% during it) so the first video preview is instant. This **must** be on the main thread — Qt multimedia objects have main-thread affinity, and offloading the load to a worker thread doesn't help (it holds the GIL; see "Startup splash" → why the bar can't animate during the load) and is risky on this stack. (History: was a deferred `QTimer.singleShot(1500, …)` warm-up to keep cold start ~0.12s; the user chose a splash + eager init. If the splash is ever removed, restore the deferred warm-up rather than blocking a bare launch ~2s.)
- **Why ffmpeg is involved at all:** only the **video** branch needs it — image (`QPixmap`) and animated-WebP (`QMovie`) previews are pure image pipeline. A video file is compressed frames, so showing anything requires *decoding*, and Qt Multimedia's decoder backend on this build is FFmpeg. The current design plays the clip live (`QMediaPlayer.play()` looping, silent — no audio output), which is what pulls in the in-process Qt FFmpeg backend (≠ the transcode path's ffmpeg *subprocess*).

#### Potential alternative: static thumbnail previews (drop Qt Multimedia from the preview pane)
Not implemented — recorded as a cleaner direction if a *still* preview is acceptable. Instead of playing the clip, decode just the **first frame** with the **bundled `ffmpeg` executable** (subprocess, exactly like `transcode.py`) and show it as a `QPixmap` on the same `QLabel` images already use. Sketch: `ffmpeg -i <src> -frames:v 1 -y <tmp.png>` (or `-ss <t>` for a frame past black intros), run via the existing `_run_ffmpeg`/`_ffmpeg_tool` helpers, ideally on a worker thread (subprocess work has no GUI-thread affinity), then set the pixmap when it returns.
- **What it removes:** the entire `QMediaPlayer`/`QVideoWidget` preview stack, `_ensure_preview_player()`, the lazy-init null-checks, the `QTimer` warm-up, and the index-1 `QStackedWidget` page — the preview pane collapses to a single `QLabel` (image / WebP `QMovie` / video thumbnail all render there). Qt Multimedia is then used **only** by `InteractiveVideoItem` on the canvas, not by the browser.
- **Why it's architecturally cleaner:** kills the ~2s GUI-thread backend init for the common case (no warm-up, no first-preview hitch, no main-thread affinity constraint to reason about), unifies all three preview kinds onto one widget/code path, and makes the preview's video handling consistent with the rest of the app's "ffmpeg = subprocess" convention.
- **Trade-off / cost:** the preview becomes a still frame, not a moving clip (no audio anyway, since previews are muted); needs a tiny thumbnail cache or a temp-file lifecycle, and a fallback placeholder when ffmpeg is missing/decode fails. Keep `transcode.py`'s `creationflags=_SUBPROCESS_FLAGS` so the windowed build doesn't flash a console.

In **`items.py`** (graphics items; talk to the window via runtime `view.window()`, never by importing `main`):

### Persistence & file layout
- **Map files are JSON** (see `arcaneatlas/assets/maps/testing.json`): an `items` array (each with `type`, `asset`, `pos`, `size`, `rot`, `z` — `z` is the stacking value, see Layers & z-ordering — and `visibleToPlayer`, see Per-item player visibility; tokens also carry `id` + optional `tokenColor`/`playerControllable`, see Tokens & LAN web sharing), grid/background settings, `fogEnabled`, and `fogReveal` (the reveal path serialized inline as a list of closed scene-coord polygons — see Fog of war). `type` is `"image"` (`InteractivePixmapItem`), `"video"` (`InteractiveVideoItem`), `"anim"` (`AnimatedItem`, an animated WebP), or `"text"` (`TextBoxItem` — no `asset`; carries `text`/`width`/`fontSize`/`textColor`/`bgColor`/`borderColor`/`borderWidth`/`bold`/`italic`/`align` instead, see Text boxes); `open_map` maps each back to its item class and `_save_to_path` picks the string by `isinstance`. No sibling fog file. Relevant methods: `open_map`, `save`, `new_map`, `_create_new_map_in`, `_save_fog_only`, `_save_to_path`.
- **Maps are always file-backed; there is no "unsaved blank map" or Save As.** `_create_new_map_in(dir)` writes a default-named (`New Map.json`, `New Map 1.json`, …) JSON immediately, so every editable map has a path. `save()` writes to `current_map_path` and is a no-op (and disabled) when none is open; a map's location/name is changed only from the browser (rename/move). The **no-map state** (`current_map_path is None`) shows a `WelcomeItem` — an ordinary **scene item** (z=99) centered on `canvas_rect`, so it appears in the player view like any other item and the GM can frame/pan the player view onto it. It's non-interactive (`setAcceptedMouseButtons(NoButton)` → never selectable/draggable even when the map is unlocked) and excluded from saves (no `asset_filename`), the canvas extent (`_content_rect` counts only `InteractivePixmapItem`/`InteractiveVideoItem`/`AnimatedItem`), the layers list, and grid z-ordering (`on_grid_above_toggled`). `MainWindow._update_map_ui_state()` is the single switch — it disables `save_action` and calls `_set_welcome_item_visible()`, which adds/removes the item by **searching the scene by type** (never holding a reference across `scene.clear()`, which deletes the C++ object). Fog is intentionally **not** special-cased: it's fully usable with no map (same code/perf as a real map) — it just isn't persisted. **Call `_update_map_ui_state()` after any change to `current_map_path`** (already hooked in `__init__`, `new_map`, `open_map`, `_save_to_path`). Dropping a file while at the welcome screen auto-creates a default map first (`Canvas.dropEvent`).
- **Unsaved-changes tracking is content-based** (`_has_unsaved_changes()`), not a sticky flag. `mark_dirty()` still sets `_dirty` as a cheap hint, but the actual "do I prompt to save?" decision compares a **`_saveable_signature()`** of the map's saved content against a baseline (`_saved_sig`) snapshotted at each clean point (`_mark_clean()` at the end of `__init__`/`new_map`/`open_map`/`_save_to_path`). The signature reuses **`_clip_entry_for_item`** (the copy/paste serializer) so it can't drift from what's persisted, and it **excludes player tokens** (which aren't saved) and fog/player-view (auto-saved on leave). Consequences: adding tokens and turning them into player tokens nets to **no unsaved change** (they vanish from the signature once their asset enters the roster), moving a player token doesn't count, and an edit undone back to the saved state stops counting — while any real saveable edit (items, grid colour/enable/above, bg colour) still prompts. `new_map`/`open_map` call `_has_unsaved_changes()` (the Yes/No/Cancel "Save current map?" dialog; Yes→`save()`→re-check→proceed). The mark-dirty hooks live on the bg/grid **widget signals** and on item drop/move-or-resize-release/rotate/remove/Delete-key (they only matter as the cheap hint now).
- **Asset references are library-relative POSIX paths, NOT bare filenames** (e.g. `"backgrounds/foo.png"`), resolved as `os.path.join(asset_dir, ref)`. **Rule: never look an asset up by basename or assume names are globally unique — two categories can each hold a `goblin.png`.** The library is `Documents/ArcaneAtlas/assets/` with category subfolders `backgrounds/`, `objects/`, `tokens/` (`ASSET_CATEGORIES`); the category for an external drop is chosen via the import dialog (see Asset categories), defaulting to `DEFAULT_DROP_CATEGORY` (`backgrounds`) when not otherwise specified. `_import_asset(src, category)` / `_import_video(src, category)` / `_import_animated_object(src, category)` copy into the category subfolder, collision-rename within it, and return the relative path; the first two no-op the copy (returning the existing relative path) when `src` already lives anywhere under `asset_dir` (prevents re-save duplication). Deferred image imports remember their chosen folder via `item.asset_category`, consumed by `_save_to_path`. Old maps that stored bare filenames keep resolving at the library root, so flat and subfoldered refs coexist with no migration of map JSONs.
- **User data lives in `Documents/ArcaneAtlas/`** (`user_data_root()` via `QStandardPaths`), with `maps/`, `assets/`, and `settings.json`. `ensure_asset_dirs()` runs at startup: it renames a legacy flat `assets-library/` → `assets/` (one-time, only if `assets/` is absent) and creates the category subfolders. **The library was renamed from `assets-library` → `assets`; the rename happens automatically on next launch.**
- **Starter content seeding**: on every launch `seed_starter_content()` copies packaged sample maps/assets from `arcaneatlas/resources/starter/` into the user folder *without overwriting existing files*. It's gated by `STARTER_VERSION` written to a `.starter_seeded` marker — **bump `STARTER_VERSION` to force a re-seed** on upgrade.

### Resource loading (dev vs frozen)
Use `res_path(name)` for any bundled resource (icons, etc.). It resolves across three modes: PyInstaller's `sys._MEIPASS`, `importlib.resources` over the `arcaneatlas.resources` package, and a dev source fallback. The `.spec` deliberately keeps `Analysis.datas` empty and ships `resources/` via a `Tree(...)` in `COLLECT` — keep resource bundling there, not in `datas`.

## Future direction: remote clients (phone / web companion)
**Partially implemented** — a first cut now ships as **LAN web sharing** (see that section): the cheap **pixel-stream** (A, below) for viewing + a minimal **state-sync** layer (B) for token dragging, LAN-only, plus the two prerequisites (mutation chokepoint `apply_token_move`, stable per-item `id`). What remains is mostly the richer end of B (browser-side scene *rendering* instead of pixels) and reach beyond the LAN (tunnel/WebRTC). The notes below stay as the design map for that.

The long-term goal is an optional phone/browser companion that can (a) view the Player View and (b) move tokens. Two viable architectures, very different cost:

- **A) Pixel-stream the Player View (cheap, view-mostly).** Capture the player window's framebuffer (`PlayerWindow.grab()` or a `QVideoSink`) and stream it — MJPEG over a tiny embedded HTTP server for a prototype, or WebRTC for low latency. **Pros:** reuses existing rendering, so fog/video/alpha all "just work"; good for a quick "see the table on a phone" win and to prove the embedded-server plumbing. **Cons:** bandwidth/latency; interaction is awkward (client only has pixels → must ship click coords back and map to scene coords; no real token model on the client).
- **B) State-sync + browser rendering (Foundry-lite, interactive).** Send scene *state* to the client and render it in the browser (PixiJS/Canvas); client sends back token-move commands. **Pros:** crisp at any resolution, low bandwidth (deltas), natural token interaction, multi-client. **Cons:** must expose state over a protocol, serve assets over HTTP, and **re-implement rendering client-side** (note: a browser decodes WebM alpha natively, so the [transparency WebP transcode](#why-transparency-uses-a-separate-pipeline-alpha-is-the-whole-reason-webp-exists-here) is a desktop-only concern — a web client wouldn't need it). This is the real path for *moving tokens*.

**What the current design already gives this for free** (don't regress these): the **single-source-of-truth shared scene** means a remote mutation applied to the scene updates GM + Player views automatically — only the remote producers/consumers are new, not local rendering; **state is already serializable** (the map JSON schema — items with `pos/size/rot`, fog as scene-coord polygons — should be reused as the wire format); **scene coordinates are well-defined** (72px = 1 inch) so touch input maps cleanly; the **player-view rect** already defines exactly the region a remote viewer should see.

**Prerequisites / cheap prep that pays off later** (the first two are worth doing even without networking, and clean up current code):
1. **A single authoritative "apply mutation" chokepoint.** Mutations are currently applied ad-hoc inside event handlers (`mouseReleaseEvent`, fog reveal/hide, remove, etc.). Funnelling them through methods like `apply_item_move(id, pos)` / `apply_fog_reveal(...)` gives the natural seam to later broadcast-to / receive-from remote clients.
2. **Stable per-item IDs.** Map items have no `id` today (identified positionally / by asset ref). A network protocol needs a stable handle per token — add a persistent `id` to each item in the map JSON.
3. **An embedded server** (Qt's `QWebSocketServer`, or a small `aiohttp`/FastAPI on a worker thread — same threading pattern as transcode) that broadcasts state deltas, accepts token-move commands, and serves assets over HTTP. At a table this is LAN-only; the usual UX is "host shows a QR code, players join the local server," so auth stays simple.

**Sequencing:** tokens now exist as static round images (see Tokens & the tokenizer) — the remaining "move tokens" prerequisites are: a central mutation layer + item IDs → embedded WebSocket/HTTP server → browser client (start view-only via state-sync, then enable token drags). A pixel-stream prototype (A) is a good early checkpoint before committing to the full client renderer (B).

## Conventions / gotchas
- **Module split is shallow and one-way** (see Module layout): `items.py`, `transcode.py`, `tokenizer.py`, and `webserver.py` must never `import main` — items reach the window via runtime `view.window()`, `webserver` via a window ref passed to its constructor. Keep new graphics items in `items.py`, ffmpeg work in `transcode.py`, networking in `webserver.py`; leave `MainWindow`/`Canvas`/`PlayerWindow` in `main.py` (don't split `MainWindow` into mixins). `main.py` is still large by design (`MainWindow` is one big class).
- `ui_mainwindow.py` is **generated** — never hand-edit it; change `mainwindow.ui` and rerun `build_ui.sh`.
- The Linux `build_ui.sh` uses `--from-imports` but the Windows `build_ui.bat` does not; regenerate on the same platform conventions to avoid spurious import diffs.
- `nondistribute_resources/` holds source art not shipped with the app.
- **Don't set the scene rect directly** (`setSceneRect(itemsBoundingRect())`) when maps change — go through `refresh_canvas_extent()` so the grid stays consistent with `canvas_rect`. `create_grid()` reads `canvas_rect`; `setSceneRect` is called *inside* it.
- **Fog reveals live in scene coordinates** (`Canvas.fog_reveal_path`). Reset via `_reset_fog_mask()`, mutate only via `reveal_at`/`hide_at`, and after changing it on the GM canvas push it to the player (`_push_fog_to_player()` / `sync_fog_to_player_view()`) — the player holds a separate path reference, not a shared object.
- **`Canvas.scene` is an attribute, not the `QGraphicsView.scene()` method** (the attribute shadows it). For the GM canvas it's the real scene; for the `PlayerWindow` canvas it's the canvas's *own* empty scene (the shared scene is attached via `setScene()`, which doesn't update the attribute). Don't rely on `self.scene` inside code that runs on the player canvas (e.g. `drawForeground`) — use scene-coordinate state like `fog_reveal_path` instead.
- **The scene uses `QGraphicsScene.NoIndex`** (set in `Canvas.__init__`). The default BSP item index caches item bounding rects and can dereference a stale entry during paint while an item's geometry is changing (a live resize calls `setScale`/`moveBy` every mouse-move), segfaulting in `QGraphicsItemPrivate::effectiveBoundingRect` with no Python traceback. Linear lookups are fine at a battle map's item count. **Don't re-enable BSP indexing** without confirming the resize/move crash doesn't return.
- **`main()` calls `_setup_diagnostics()` (crash/exception logging) — keep it.** It enables `faulthandler` so native/Qt segfaults dump a Python stack instead of vanishing. To capture a crash on dev/console runs, run `./runlin.sh 2>&1 | tee /tmp/aa-crash.log` and read the tail. **The windowed (`console=False`) PyInstaller build has no console — `sys.stdout`/`sys.stderr` are `None`** — so `_setup_diagnostics()` routes diagnostics to `Documents/ArcaneAtlas/logs/` instead: native faults → `crash.log` (via `faulthandler.enable(file=…)`; calling it on `None` stderr is what raised `RuntimeError: sys.stderr is None`), and uncaught Python exceptions + stray `print()`s → `app.log` (it redirects `stdout`/`stderr` to the file, so the **default** excepthook captures otherwise-invisible exceptions — no custom hook). Both logs are size-capped: `_trim_log()` rolls each over to a single `.1` backup at startup once it passes ~1 MB, so they can't pile up across launches. All of it is best-effort and never blocks startup. **After an installed-app crash, look in `Documents/ArcaneAtlas/logs/`.**
- **Runtime — standardizing on Python 3.13.** The dev `.venv` is pinned to **Python 3.13 + PySide6/shiboken**, moved off bleeding-edge **3.14**, which produced intermittent native crashes in the Qt scene/index/paint or multimedia paths (partly timing/refcount-sensitive). **Shipped builds were already on stable 3.12** (see `release.yml`) — so those crashes were dev-only. **CI/build deliberately stays on 3.12 for now**; the `release.yml` bump to 3.13 is deferred to a verified release run (rationale in the *remote online play* roadmap's Python-3.13 decision). Dev-3.13/build-3.12 is a tiny adjacent-version gap. **Keep the teardown discipline regardless of version:** video items stop their player before teardown (`InteractiveVideoItem.teardown_player()`, called on Remove, on Delete-key, and before every `scene.clear()` via `_stop_all_videos()`) to avoid the FFmpeg backend writing to a freed sink. `AnimatedItem` exposes the same `teardown_player()` (stops its `QMovie`) and `_stop_all_videos`/the Delete handler treat both types together — so any new media-backed item should follow that `teardown_player()` convention. If native crashes still appear on 3.13, they're app logic or a shiboken issue — not a 3.14 artifact (3.13 removes that variable).
- **Video looping uses the backend, not a manual seek.** `InteractiveVideoItem` sets `player.setLoops(QMediaPlayer.Infinite)`. **Don't reintroduce the old `positionChanged → setPosition(0)` loop** — it never re-issued `play()`, so if a clip reached `EndOfMedia` (→ Stopped) before the near-end position tick fired, the frame went **blank** (intermittent; worse with several concurrent videos competing for CPU during zoom/pan). The item also logs `errorOccurred`/`mediaStatusChanged` (to `app.log` via the `arcaneatlas.items` logger) and re-issues `play()` on an unexpected `EndOfMedia`. A blanked video with the item box still present is the signature of this class of multimedia-backend failure — check `Documents/ArcaneAtlas/logs/app.log` for `Video playback error` / `EndOfMedia`. Running several FFmpeg decoders at once is a real resource ceiling; if logs show genuine backend errors, the fix is fewer concurrent *playing* videos (e.g. still-frame when off the player view), not more loop hacks.

## Future direction: remote online play + embedded web companion (vgbnd) — NOT DONE (roadmap)

**Status: NOT DONE — design/roadmap only, nothing here is implemented.** This is the concrete continuation of *Future direction: remote clients* (Option A, the pixel-stream) and *LAN web sharing* — reusing their already-built prerequisites (`apply_token_move` chokepoint, stable per-token `id`s). It is a **collaboration** with the **vgbnd** ttrpg game-companion web app.

### The thesis (who owns what)
Two mature products get married along a clean seam: **ArcaneAtlas owns mapping + the desktop VTT** (local map/token hosting, GM UI, only the GM installs it); **the vgbnd web app owns all game-system knowledge** (text chat, OpenGL 3D dice — swappable per system to coins / a Dread jenga tower / etc. — character sheets, brand resources; Firestore-backed, account-scoped). They join at two points: (a) Atlas streams the **player view** out to remote browsers; (b) vgbnd supplies the **right-side companion UI** — seen by remote players *in their browser* and by the GM *embedded inside Atlas*. **Remote play needs no router port forwarding** because the GM app dials **outbound** to a vgbnd-hosted relay.

### Architecture
- **Atlas's online-path job is only two things:** stream the **player-view video** out (**WebRTC** — the chosen transport; see the transport decision) and carry **token state + moves** (the existing `apply_token_move` seam, `main.py`). Nothing else.
- **vgbnd owns chat / dice / character / resources** entirely (Firestore; persistence survives sessions but isn't mandatory). Game-system-swappable.
- **GM surface:** Atlas renders the map natively and **embeds the vgbnd companion panel via `QWebEngineView`** — a docked right pane on the existing `self.splitter` (built in `MainWindow.__init__`) + a **pop-out** button. The GM does *all* interaction through Atlas.
- **Remote player surface:** one vgbnd browser page — Atlas's player-view video on the left (via the relay), vgbnd's companion UI on the right.
- **Standalone display — NO Atlas→web data bridge.** Atlas only *hosts/streams*; it pushes nothing into the web app (join/move/presence are the relay's/vgbnd's concern; chat/dice/character live wholly in vgbnd + Firestore). So **no `QWebChannel`, and no native chat/dice code in Atlas** (an earlier native-panel design was explicitly dropped — reimplementing vgbnd in Qt would duplicate it, lose game-system swappability, and grow the ttrpg knowledge the split deliberately keeps web-side).
- **Local/LAN play is UNCHANGED and stays offline-capable** — the built-in `INDEX_HTML` LAN page (map + token control, see *LAN web sharing*) is retained for same-room tablets. Chat/dice/character is an **online-session** surface (GM + remote players), not a local-tablet feature (local players are at the table). *(Open assumption — revisit if local tablets should also load the vgbnd page, which would require internet even for local play.)*

### Key decisions (settled in planning)
- **Servers — two pieces.** **vgbnd** (the friend's) hosts the **production** relay + the web companion (chat/dice/character, Firestore). **ArcaneServer** (the user's **private** repo, hostable locally on Linux) is the **dev/verification webserver** for building and proving the WebRTC pipeline standalone (signaling now; media/SFU as it grows) and doubles as the interchangeable **local mock relay**. **vgbnd's wire protocol is the source of truth** (`packages/shared/src/protocol`, `t`-discriminator JSON: `gm_hello`/`player_hello`/`hello_ack`/`tokens`/`move`/`presence`/`session_end`) — both ArcaneServer and Atlas's messages conform to it, so ArcaneServer stays drop-in interchangeable with vgbnd's production relay. *(Open coordination point with the friend: whether ArcaneServer is purely a dev stand-in that conforms to vgbnd, or the user's own contributed server that vgbnd adopts.)*
- **Video transport: WebRTC first (verified standalone), MJPEG as optional fallback.** *(Reversed from an earlier MJPEG-first lean once the stack was hardened to 3.13.)* Build and prove a **WebRTC pipeline first — independent of the current local player-control path** — against **ArcaneServer**, then wire it into the `remoteplay` fork. Rationale: hardening to **3.13** removes the main reason WebRTC was being deferred (aiortc/PyAV bundling fragility on 3.14); WebRTC is the better long-term transport (smooth, adaptive, low-latency) and **cheaper at scale** (an SFU sends one compressed stream vs. MJPEG's fat frames × fps × players); and doing it **standalone** de-risks it without destabilizing the working local player control. **MJPEG stays** as the existing LAN page (unchanged) and a possible remote **fallback** if WebRTC proves too fragile/costly on a given path — with **dirty-frame skipping** + low-fps as cheap wins there. *(Cost context that informed this: MJPEG bandwidth is real — ~4 GB/player/hour at ~8fps — trivially cheap on a bandwidth-included VPS but it scales with fps×players; WebRTC-SFU is ~5–10× less egress but carries build/ops overhead. So MJPEG is cheaper small/early, SFU cheaper at scale.)*
- **Companion panel = embedded vgbnd via `QWebEngineView`** loading a vgbnd **panel route** with the session/room code; lazy-constructed only in online mode.
- **Remote join: room code only** for now. Upgrading later to vgbnd's pairing-code + Firestore/accounts is a *small* Atlas change (obtain a code string → put it in the handshake + the panel URL; all validation lives in vgbnd).
- **Dev strategy — FORK named `remoteplay`.** Build the web-integrated / remote-play Atlas on the **`remoteplay` fork**; keep normal bug-fix/feature work on `main`; **merge back only once fragility + function are proven**.
- **Python 3.13 for the whole project (in progress).** Standardize **all three apps** (`main` + the `remoteplay` fork) on **3.13**. **Key finding:** the crash-prone **3.14** was only the local dev `.venv`; **shipped builds were already on stable 3.12** (`release.yml`) — so those crashes were dev-only. The move is (a) dev `.venv` 3.14→3.13 and (b) an optional CI bump 3.12→3.13. **Done:** `requirements.txt` setup notes, `.python-version` files, READMEs, and the runtime notes across ArcaneAtlas/Eye/Audio. **Pending:** the `release.yml` CI bump 3.12→3.13 (touches the macOS universal2 python.org `3.12.8` pkg + `/Versions/3.12/` paths and ArcaneAudio's `--python-version 3.12 --abi cp312` wheel-fusion — mechanical but verifiable only by a real release run) and recreating each local `.venv` on 3.13. **3.12 rejected** — marginal extra soak for ~a year less support runway; drop to it only if a dep ever lacks a 3.13 wheel (none does).

### Fragility strategy (Chromium on the PySide6/shiboken/PyInstaller stack)
`QWebEngineView` ships Chromium (~100+ MB, `QtWebEngineProcess`/ICU/locales). Mitigations, priority order: **(1)** the Python-3.13 pin above (highest-leverage); **(2)** QtWebEngine runs Chromium **out-of-process** (sandboxed GPU/renderer) — only the thin widget is in-process, so the added in-process native surface is modest and a renderer crash needn't kill Atlas; **(3)** **lazy + optional init** — the web view spins up only in online mode, so a bare/offline launch never touches QtWebEngine (blast radius contained, existing local path unchanged); **(4)** pin the exact PySide6 version (QtWebEngine + Multimedia known-good together) and add the web build to CI early; **(5)** **macOS caveat** — Chromium's several helper executables each need signing, complicating the existing notarization step (budget for it); **(6)** the fork itself de-risks `main`.

### Technical notes to preserve
- **MJPEG dirty-frame skipping:** hash/signature the last grabbed frame; skip the send when the player view is unchanged (idle bandwidth ≈ 0). A "view changed" hint can come from scene/fog/player-rect edits or a cheap image compare.
- **Token control stays on the control channel** (relay WS / LAN WS), fully decoupled from video — so the eventual MJPEG→WebRTC swap doesn't touch it.
- **Remote delta from local is small:** control transport flips from the inbound `QWebSocketServer` to an **outbound `QWebSocket` client**; the relay fans frames/state out and forwards `move` back. Keep envelopes conforming to vgbnd's protocol so the pipeline is transport-agnostic.
- **WebRTC pipeline notes (the primary path):** GUI-thread grab → RGB888 ndarray with **even W/H** (VP8/H.264 requirement) → single latest-frame slot (drops stale frames, GUI never blocks); browser `<video>` sized to **intrinsic px, NOT `object-fit: contain`** (letterbox bars break the `nx*width` token-overlay math — see *LAN web sharing*'s normalized-rect scheme); SDP/ICE over the signaling WS (ArcaneServer/vgbnd); `iceServers` empty on LAN, gains STUN+TURN (coturn) for remote NAT traversal.
